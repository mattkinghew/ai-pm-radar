# IP Building Workflow

## Purpose

This document defines a documentation-first workflow for turning research, project notes, interviews, course materials, activity records, AI PM Radar articles, screenshots, PDFs, and transcripts into reusable IP content and portfolio evidence.

The workflow is designed for AI PM / digital transformation consultant positioning. It supports content creation, campaign planning, landing page thinking, and case study generation without exposing private client data or overclaiming results.

## Workflow Overview

```text
Raw input → Clean source note → AI-assisted synthesis → Human review → Reusable output → Portfolio evidence
```

The goal is not to publish every input directly. The goal is to extract reusable insight, decision logic, audience pain points, and practical examples.

## Input Types

| Input Type | Example | Main Value | Handling Notes |
|---|---|---|---|
| Long-form notes | Obsidian / Notion research notes | Deep ideas, frameworks, observations | Summarize and remove private details before AI processing |
| Client interviews | Discovery call notes, stakeholder comments | Pain points, vocabulary, buying objections | Anonymize client identity and sensitive business context |
| Course materials | AI, PM, marketing, cloud, security notes | Structured knowledge and terminology | Avoid copying paid content verbatim |
| Activity reports | NGO / education / campaign reports | Real project process and impact logic | Remove participant names and personal data |
| Product research | Tool comparisons, competitor notes | Product angle, market positioning, user needs | Track source URL and date |
| AI PM Radar articles | Daily radar entries | Source-aware AI PM insight | Reuse summary, business angle, AI PM angle, and risk note |
| Screenshots | Product UI, campaign examples, analytics view | Visual evidence and UX observations | Blur private data before analysis |
| PDFs | Reports, white papers, decks, manuals | Long-form source material | Extract only relevant sections and cite source internally |
| Transcripts | Meetings, interviews, workshops, voice notes | Natural language pain points and decisions | Review speaker attribution manually |

## Processing Steps

### 1. Capture Source

Record the minimum metadata before processing:

```text
Source title:
Source type:
Source URL / file location:
Date captured:
Original context:
Why it may be useful:
Privacy level: public / internal / sensitive
```

### 2. Clean and Reduce Risk

Before using AI tools:

- remove names, phone numbers, emails, addresses, IDs, account numbers, and confidential financial data
- replace client names with generic labels such as `Client A`, `SME retailer`, or `NGO programme team`
- avoid pasting raw private Obsidian vault content
- keep only the excerpt needed for the specific task
- mark assumptions clearly

### 3. Extract Reusable Insight

For each source, extract:

```text
Core idea:
Audience pain point:
Useful quote or wording pattern:
Business / operation implication:
AI PM implication:
Marketing angle:
Risk or limitation:
Possible output format:
```

### 4. Convert into Output Candidates

Turn the cleaned insight into one or more reusable outputs:

- Threads posts
- IG captions
- LinkedIn posts
- carousel outlines
- campaign ideas
- landing page angles
- portfolio case notes

### 5. Human Review

Before publishing or adding to portfolio:

- confirm facts against original source
- remove unsupported claims
- check whether the content sounds like advice, guarantee, diagnosis, legal opinion, or financial recommendation
- check whether sensitive context can identify a real client or participant
- confirm tone is useful, not hype-driven
- preserve source links or internal source references where needed

## Human Review Rules

1. Do not publish AI-generated content without source review.
2. Do not present draft ideas as proven results.
3. Do not imply real client impact unless there is verified evidence.
4. Do not copy course, PDF, or newsletter content verbatim.
5. Do not use private meeting notes as public content unless they are anonymized and generalized.
6. Always separate observed facts, interpretation, and recommendation.
7. Keep a risk note for AI, data, privacy, or stakeholder misunderstanding.

## Privacy Rules

### Never include

- API keys, `.env`, credentials, passwords, tokens, private keys
- raw client documents
- personal data of clients, students, participants, or service users
- confidential business strategy or pricing
- private Obsidian vault content copied in bulk
- screenshots containing names, emails, phone numbers, account IDs, or private dashboards

### Safer replacement pattern

```text
Original: A named client in the beauty sector asked us to improve their campaign conversion after a failed launch.
Safer: An SME service team wanted to improve campaign clarity after previous content did not convert well.
```

## Example Workflow

### Input

A PDF report about AI adoption in SMEs and one AI PM Radar article about AI workflow governance.

### Clean Source Note

```text
Source type: Public PDF + AI PM Radar article
Core idea: SME AI adoption fails when tools are added without workflow redesign.
Audience: SME owners, NGO managers, education programme leads
Pain point: Teams want AI efficiency but lack governance and review process.
Risk: Over-automation may expose private data or create inaccurate client-facing content.
```

### AI-Assisted Synthesis

Possible angles:

- Threads: “AI adoption is not a tool list. It is a workflow redesign problem.”
- IG: Simple checklist for safe AI use in small teams
- LinkedIn: Reflection on human-in-the-loop AI PM workflow
- Carousel: 7-step client-safe AI workflow
- Landing page: AI Workflow Audit for small teams
- Portfolio note: Documentation-first workflow design for AI adoption

### Human Review

Check:

- whether the SME adoption claim is supported by the source
- whether the example is generic enough
- whether the risk note is clear
- whether the CTA avoids overpromising business results

## How Installed Skills Support This Workflow

### pdf

Use for extracting and reviewing long PDF reports, course notes, white papers, and proposal documents. The output should become structured source notes, not copied content.

Recommended use:

```text
PDF → section summary → source note → content angle → human review
```

### screenshot

Use for analyzing UI examples, product flows, campaign layouts, landing pages, or dashboard screenshots.

Recommended use:

```text
Screenshot → visual observation → UX / marketing insight → anonymized example
```

Always blur or crop private information before analysis.

### transcribe

Use for converting interviews, meetings, voice notes, workshop recordings, or personal reflections into text.

Recommended use:

```text
Audio → transcript → pain points / decisions / quotes → anonymized content brief
```

Manual review is required because transcription can mishear names, technical terms, or speaker attribution.

### notion-knowledge-capture

Use for turning conversations, decisions, and learning notes into structured Notion pages.

Recommended use:

```text
Raw notes → Notion knowledge entry → reusable framework → portfolio reference
```

### notion-meeting-intelligence

Use for preparing agendas, extracting action items, and summarizing meeting context.

Recommended use:

```text
Meeting notes → action items → risk list → follow-up plan → campaign / prototype decision
```

### notion-research-documentation

Use for synthesizing multiple Notion notes into briefs, comparisons, and documentation.

Recommended use:

```text
Multiple notes → research brief → decision summary → content / case study artifact
```
