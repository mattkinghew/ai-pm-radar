# AI PM Radar Portfolio Positioning

## Purpose

This document explains how to present AI PM Radar as a portfolio-ready AI Product Manager workflow without overstating automation or engineering complexity.

## One-line Positioning

AI PM Radar is a static, human-reviewed AI intelligence workflow that turns selected daily AI news into structured product-management insight for non-technical AI PM learners, NGO and education practitioners, and SME decision-makers.

## What This Project Demonstrates

### Product thinking

- clear target users
- defined user problem
- MVP scope control
- non-goals that prevent overbuilding
- decision-oriented content structure

### AI workflow design

- AI-assisted drafting with human review
- source-aware content schema
- repeatable daily review workflow
- conservative handling of risk and uncertainty

### Portfolio value

- shows how to turn noisy AI news into actionable PM interpretation
- shows how to bridge product, operations, and lightweight technical implementation
- shows honest limits instead of pretending the workflow is fully autonomous

## Safe Claims

Use claims like:

- `Built a static AI PM intelligence MVP with structured JSON content and validation gates.`
- `Designed a human-in-the-loop content workflow for turning AI news into PM insight.`
- `Defined scoring and review logic to prioritize relevance, actionability, and portfolio value.`
- `Used low-cost architecture to validate the information product before adding backend complexity.`

## Claims to Avoid

Do not claim:

- fully automated AI news pipeline
- autonomous editorial system
- production-grade scraping infrastructure
- real-time intelligence engine
- backend content platform

unless those features actually exist and are verified.

## Recommended Narrative

Use this structure in interview, README, or case study explanations:

### 1. Problem

AI news is abundant, but non-technical readers often struggle to translate headlines into product decisions, workflow changes, or safe adoption practices.

### 2. Product decision

Start with a static MVP and structured JSON model instead of scraping, backend systems, or account-based personalization.

### 3. Workflow design

Use a manual or semi-manual pipeline:

- collect sources
- draft structured entries
- review for accuracy and risk
- validate JSON
- publish static output

### 4. Why it matters

This proves the product logic first:

- Does the format help users interpret AI news?
- Can the workflow stay maintainable for a solo builder?
- Can the project demonstrate AI PM judgment clearly?

## Interview Talking Points

### Why not automate everything?

Because the first product question is whether structured interpretation is useful, not whether a crawler can collect more content.

### Why keep human review?

Because source quality, actionability, Hong Kong relevance, and portfolio value require judgment. AI assistance can speed up drafting, but the final decision should stay human-reviewed.

### Why is this an AI PM project, not just a content site?

Because the core work is:

- defining the user problem
- shaping a repeatable decision format
- creating scoring logic
- designing review checkpoints
- controlling scope and risk

## Portfolio Boundaries

Be explicit that the current version is:

- static
- local-data driven
- validation-backed
- manual or semi-manual
- beginner-friendly

Be explicit that the current version is not:

- a scraping platform
- a CMS
- a recommendation engine
- a fully automated content operation
